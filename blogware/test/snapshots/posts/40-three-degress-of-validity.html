Three degrees of validity

✑2025-04-26

  • An example
  • Practical implications
  • Appendix: facts and theorems

This article presents a view of system configuration as a program for an embedded interpreter. This view leads to three logical processing stages: parsing, analysis, and execution. As the configuration moves through that pipeline, our confidence in its validity increases.

Configurations as programs

All configuration files are programs for an interpreter embedded in a larger system. These programs encode the system initialization steps or supply data for them. The most widely used configuration formats are declarative (xml, json, yaml, toml), but some systems implement functional (Google’s GCL, Dhall, nix, Starlark) or imperative (VimL, Emacs Lisp) configurations. The configuration interpreter is rarely a specific module; more often, it’s a cross-cutting concern spanning the initialization logic of other system components.

In a typical interpreter, a program goes through three conceptual stages:

  1. Parsing. The interpreter recognizes the program as belonging to the language it understands, extracting a syntax tree from the input. The xml standard calls documents passing this stage well-formed.
  2. Static analysis. The interpreter executes the program symbolically and detects internal inconsistencies. In configuration terms, this step matches the parsed configuration object against a schema. The xml standard calls documents passing this stage valid.
  3. Execution. The interpreter runs the program. In configuration terms, this stage usually requires the interpreter to interact with its environment (bind sockets, connect to databases, etc) to initialize the system components. We’ll call configurations that initialize the system correctly applicable.

The stages are conceptual because interpreters can transition between them freely. They can parse a document section, validate it, execute the corresponding effects, and proceed to parse the next section.

Configuration systems that support external references For example, Dhall supports including the contents of an arbitrary url into the document, xml supports this feature through XInclude specification and ENTITY declarations, and nix supports import expressions and raw file inclusion and provides a collection of fetchers for various external resources. rely on interleaving the parsing stage with the effects of reading external files. These effects are benign and usually confined to the parser, so this feature doesn’t invalidate the conceptual model.

An example

Let’s consider an example: a TOML configuration file of a typical backend service that connects to a database and starts an HTTP server.

db_url = "postgres://user:password@host:5432/dbname"
http_port = 8080

TOML’s top-level data structure is a hash table, so the parsing stage outputs a table with two entries.

Table({
  "db_url": String("postgres://user:password@host:5432/dbname"),
  "http_port": Integer(8080),
})

The TOML parser doesn’t understand the field semantics. It would just as happily decode a configuration with the field values swapped.

Table({
  "db_url": Integer(8080),
  "http_port": String("postgres://user:password@host:5432/dbname")
})

The static analysis phase ensures the configuration object has the expected shape: The db_url value must be a valid URL, the http_port value must be an integer between 0 and 65535, and the table must contain no other keys. In Rust, the serde library fuses parsing and static analysis by leveraging Rust’s type system as its schema language.

#[derive(serde::Serialize, serde::Deserialize)]
#[serde(deny_unknown_fields)]
struct Config {
    db_url: url::Url,
    http_port: u16,
}

The execution phase creates and binds a socket and opens a database connection. For these steps to succeed, the following assumptions must hold:

  • The url points to a running service that understands the Postgres protocol and accepts the credentials.
  • No other process on the system uses the port.

The impossibility of checking such assumptions without system calls draws the line between validity and applicability: Validation treats the configuration as a self-contained mathematical object and relies only on rules independent of the world state, while checking applicability might cause side effects. Applicability, unlike validity, is a fleeting property: it holds only on a specific machine and at a particular time.

Practical implications

The primary benefit of the three-stage view is its nomenclature, especially its distinction between validity and applicability.

Configuration-heavy programs can benefit from making the three stages more explicit. Localizing validation in a pure function enables a trivial implementation of the --dry-run feature that validates the configuration without side effects.

Most systems I worked with extracted the execution step in an initialization routine that instantiated and assembled the system components. However, complex software targeting non-expert users can benefit from a debugging execution routine that detects applicability issues without causing irreversible side effects. Internal Google tools often offer the --diagnose_me feature that examines external dependencies and reports whether they function correctly. The tool output can help users fix their environment or serve as a basis for bug reports.

Appendix: facts and theorems

The three-stage lens also parallels our treatment of natural languages and logic: Sentences and propositions can be invalid in at least three ways.

When we process a statement in a natural language, we must first recognize it as a grammatical construct, then assign a meaning to it The Jabberwocky poem provides examples of grammatically correct but meaningless sentences, such as all mimsy were the borogoves. The sentence structure implies that every borogove in a group had the property of being mimsy. Since we lack this statement’s meaning, we couldn’t know whether it was true even if we had witnessed the scene. , and finally evaluate its correctness by engaging with the world or using our experience.

Interaction with the world distinguishes plausible statements from facts. For example, checking that all swans are white requires examining every swan or finding a pigmented one. Furthermore, many facts are transient: it might be that all swans used to be white until a mutation occurred, or they might all become white in the future if the pigmented ones become extinct.

Mathematical statements go through the same three-step process: we (1) recognize a sequence of tokens as a formula, (2) recognize it as a proposition by understanding its meaning, and (3) search for its proof to promote it to a theorem. The third step is again the most laborious, but it doesn’t require interacting with the world since the mathematical universe is independent of reality. Proving theorems feels deeply satisfying because it uncovers eternal truths. The world can’t take them away.
